Your dotfiles.
Your secrets. Managed.

A minimal dotfiles manager that turns config files into Handlebars templates, injects secrets from your password manager at render time, and keeps everything synced via git — without ever committing a real secret.

Core mechanic

From template to home directory

dotf sits between your git repo and your home directory. Templates live in git. Secrets never do. Rendered files live on disk, gitignored, reachable via symlinks.

📄 .gitconfig.tmpl → lives in ~/dotfiles/configs/ • committed to git • contains {{GIT_EMAIL}}
+ 🔑 .secrets.toml → GIT_EMAIL = "op://personal/github/email"
↓ dotf fetches secret, injects into template
⚙ .gitconfig → rendered file in ~/dotfiles/configs/ • gitignored • real email injected
↓ symlink created
🏠 ~/.gitconfig → symlink pointing at rendered file • your apps see a normal file
1

Add a config file

Run dotf config ~/.gitconfig. dotf shows the file contents and walks you through replacing sensitive values with placeholders like {{GIT_EMAIL}}. The template is written back to ~/dotfiles/configs/.gitconfig.tmpl.

2

Map placeholders to secrets

dotf writes entries to ~/.secrets.toml mapping each placeholder to a URI like op://vault/item/field or pass://vault/item/field. You choose which backend per secret.

3

Render and symlink

dotf fetches each secret from your password manager, renders the template, writes the output to ~/dotfiles/configs/.gitconfig (gitignored), and creates a symlink at ~/.gitconfig.

4

Sync across machines

Run dotf sync to pull the latest templates, re-render all configs with fresh secrets, and push any local changes. On a new machine, dotf init does it all from scratch.

~/dotfiles/configs/.gitconfig.tmpl

[user] email = {{GIT_EMAIL}} name = Chris [core] editor = nvim pager = delta [github] token = {{GITHUB_TOKEN}}

~/dotfiles/.secrets.toml

# Placeholder → secret URI mapping [secrets] GIT_EMAIL = "op://personal/github/email" GITHUB_TOKEN = "op://personal/github/token" SSH_KEY_PASS = "pass://ssh/main/passphrase" AWS_KEY = "bw://aws-root/access-key" NPM_TOKEN = "env://NPM_AUTH_TOKEN"

Reference

Command reference

All commands. No magic flags required — dotf is meant to be obvious.

Setup
dotf init
First-time setup on a new machine. Clones (or creates) your dotfiles repo, checks that secret backend CLIs are installed, runs your Brewfile, and installs shell completions.
Day-to-day
dotf config <path>
Add a config file to be managed. Shows the file, walks you through replacing secrets with {{PLACEHOLDER}} tokens interactively, writes the template, and creates the symlink.
dotf modify [name]
Opens a managed config's template in $EDITOR. Re-renders automatically on save so you can verify the output immediately.
dotf sync
git pull --rebase → fetch all secrets → re-render all templates → git push. The one command you run to keep everything up to date.
dotf diff [name]
Shows a diff between the current rendered config and a fresh re-render. Useful for spotting template drift before committing.
dotf status
Colored table of every managed config: template path, symlink target, secret backend, and health (ok / missing / broken).
dotf remove [name]
Stop managing a config. Optionally restores the rendered file in-place so nothing breaks after removal.
Secrets
dotf secrets list
List all placeholder → URI mappings from .secrets.toml.
dotf secrets validate
Attempt to fetch every secret and report which ones fail — before a sync surprises you.
dotf secrets add
Interactively add a new placeholder → URI mapping.
dotf secrets remove
Remove a placeholder mapping. Does not delete the secret from your password manager.
Utilities
dotf completions <shell>
Generate shell completions for bash, zsh, or fish. Run once and pipe to your completions directory.

Pluggable

Secret backends

Each placeholder maps to a URI that tells dotf where to fetch the value. Mix and match backends — your SSH passphrase can come from Proton Pass while your AWS key comes from 1Password.

🔒
Proton Pass
Uses the pass CLI. Vaults, items, and fields map directly to the URI path.
pass://vault/item/field
Example: pass://personal/github/email
🔑
1Password
Uses the op CLI. Follows the native 1Password secret reference format.
op://vault/item/field
Example: op://work/aws/secret-key
🛡️
Bitwarden
Uses the bw CLI. Item name and field path form the URI.
bw://item-name/field
Example: bw://github-account/password
⚡
Environment Variable
Read directly from the environment. Great for CI/CD or secrets already in your shell.
env://VAR_NAME
Example: env://NPM_AUTH_TOKEN

Switching password managers is painless

All your secret URIs live in one file: .secrets.toml. Moving from Bitwarden to 1Password means updating URIs in that file — not hunting through dozens of shell scripts and config files. Run dotf sync and you're done.

Honest comparison

Why dotf?

Existing dotfile managers are great tools. dotf is for people who want something simpler with secrets as a first-class concern.

chezmoi / yadm / stow

  • ✗ Powerful but steep learning curve — new concepts, new file structures, new commands
  • ✗ Secret management bolted on after the fact — requires chezmoi templates syntax, or external scripts
  • ✗ Switching secret backends requires reworking templates across all configs
  • ✗ Stow has no secret support at all — you manage that yourself
  • ~ All are battle-tested with large communities — right choice for complex setups

dotf

  • ✓ Conceptual model is just: git + templates + symlinks. Nothing novel to learn
  • ✓ Secrets are first-class — every placeholder maps explicitly to a typed URI
  • ✓ Swap password managers by editing one TOML file. No templates change
  • ✓ dotf init bootstraps an entire machine in a single command
  • ✓ dotf status gives you instant visibility into symlink health
  • ✓ Zero config files of its own — your dotfiles repo is the config
  • ~ Young project — production use at your own risk. Community contributions welcome.

The secret injection problem

Every dotfile manager eventually faces the same question: how do you commit a config that contains an API key without committing the key? Most answers involve encryption (age, GPG) — which adds key management overhead — or manual exclusion, which means secrets live unmanaged on disk. dotf's answer: never commit real values. Templates hold placeholders, secrets are fetched at render time from tools you already use.

Get started

Up and running in minutes

New machine or first time? Here's the full flow from zero to synced dotfiles.

1

Install dotf via Homebrew

$ brew tap chrisfentiman/hometaps $ brew install dotf
2

Initialize — first time or new machine

$ dotf init # clones your dotfiles repo, runs Brewfile, wires up completions
3

Add your first config file

$ dotf config ~/.gitconfig # interactive: highlight secrets → assign placeholders → pick backend
4

Check status and sync

$ dotf status # health check $ dotf sync # pull → render → push
5

On every subsequent new machine

$ brew tap chrisfentiman/hometaps && brew install dotf && dotf init # that's literally it
$ dotf
