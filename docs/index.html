dotf · dotfiles manager · written in Rust

Templates in git.
Secrets in your password manager.
dotf renders both into place.

dotf replaces secret values in your config files with {{PLACEHOLDERS}} that map to URIs in your password manager. At sync time it fetches and injects them. Git never sees the values.

without dotf

~/.gitconfig committed to git
[user] name = Chris Fentiman email = chris@example.com [github] token = ghp_A1B2C3D4E5F6G7H8 # ↑ real values — now in git history

with dotf

~/dotfiles/configs/.gitconfig.tmpl safe to commit
[user] name = Chris Fentiman email = {{GIT_EMAIL}} [github] token = {{GITHUB_TOKEN}}
~/dotfiles/.secrets.toml safe to commit
[secrets] GIT_EMAIL = "op://personal/github/email" GITHUB_TOKEN = "op://personal/github/token"

How it works

Three commands cover the full lifecycle.

01

dotf config ~/.gitconfig

Convert

Shows you the file contents. You identify the secret values. dotf replaces them with {{PLACEHOLDERS}}, writes the template, and asks for the password manager URI for each one.

02

dotf sync

Sync

Fetches each secret from your password manager. Renders all templates. Writes the real files (gitignored). Symlinks them into place. Commits and pushes the templates.

03

dotf init

Bootstrap

New machine. Clones your dotfiles repo. Reads .secrets.toml to check which password manager CLIs are needed. Renders everything. Done.

Install

Homebrew. Two commands.

macOS. Requires a supported password manager CLI — install whichever one you use before running dotf init.

$ brew tap chrisfentiman/hometaps
$ brew install dotf

Secret backends

Works with whatever you already use.

dotf routes each secret by URI scheme. You can mix backends in the same .secrets.toml.

Proton Pass

pass item get <uri> --fields password

pass://vault/item/field

1Password

op read <uri>

op://vault/item/field

Bitwarden

bw get <field> <item> — requires BW_SESSION env var

bw://item-name/field

Environment variable

reads from process environment — useful in CI

env://VAR_NAME

vs. the alternatives

Other tools exist. Here's the difference.

chezmoi

chezmoi has secret integration. It also has its own source directory layout, its own template dialect (Go templates), and its own config format. You're adopting chezmoi's model of how dotfiles should work, not layering on top of yours. dotf has one config file (.secrets.toml) and uses Handlebars. Your dotfiles directory stays a normal git repo.

yadm

yadm encrypts whole files with GPG. You can't encrypt just the email field in a .gitconfig — you encrypt the whole file or you don't encrypt it. That means the unencrypted version never exists in your repo, which creates friction on every new machine. dotf doesn't encrypt anything. Your password manager already does that. dotf just asks it at the right moment.

GNU stow

stow is pure symlink management. It's elegant and it works. It has nothing to say about secrets. If your configs have no secret values, use stow — it's simpler. If they do, you need something between the raw file and git. dotf is essentially stow with a template+inject layer and git sync built in.
