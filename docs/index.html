The problem

Secrets don't belong in git.

But your dotfiles need them. dotf puts a template between your config and your repo.

The old way

~/.gitconfig committed to git 
[user]
  name  = Chris Fentiman
  email = chris@example.com
  # ↑ real email, public forever

[github]
  token = ghp_AbCdEfGhIjKlMnOp
  # ↑ leaked. rotated again.

The dotf way

~/.gitconfig.tmpl safe to commit 
[user]
  name  = Chris Fentiman
  email = {{GIT_EMAIL}}

[github]
  token = {{GITHUB_TOKEN}}
.secrets.toml maps → password manager 
GIT_EMAIL    = "pass://personal/github/email"
GITHUB_TOKEN = "op://personal/github/token"

How it works

Three commands, any machine.

01

Convert

Converts your config into a template, replacing secret values with named placeholders.

$ dotf config ~/.gitconfig

02

Sync

Fetches secrets from your password manager, renders templates, symlinks the results. Commits templates — never secrets.

$ dotf sync

03

Bootstrap

New machine? One command clones your dotfiles repo, authenticates your password manager, and renders everything.

$ dotf init

Why we built it

Dotfiles are solved. Secrets aren't.

Every dotfiles manager assumes you're comfortable committing secrets to git, or makes you learn a new encryption scheme. We weren't, and we didn't want to.

01 / THE ITCH

We kept leaking secrets.

Not maliciously — just carelessly. A new machine setup, copy-paste a .gitconfig, commit it. Email address in git history forever. API token in a public gist. It was death by a thousand small mistakes.

02 / THE GAP

Existing tools made it hard.

chezmoi has secret templating, but the config format is its own language to learn. yadm encrypts whole files with GPG — great for files that are entirely secret, useless for a .gitconfig that's 95% public. We wanted something that understood the difference between a config and a secret.

03 / THE PRINCIPLE

Your password manager already has the secrets.

1Password, Proton Pass, Bitwarden — you're already paying for one, already trusting it, already using it. dotf doesn't store secrets, doesn't encrypt them, doesn't manage them. It just asks your password manager at the right moment and puts the answer in the right place.

vs. the alternatives

Good tools exist. dotf fills a specific gap.

These are all solid projects. The question is whether their tradeoffs match yours.

chezmoi

The most complete dotfiles manager out there. Templating, secret integration, cross-platform. The tradeoff: it's a system you have to learn. chezmoi-specific commands, its own source directory layout, its own template dialect. If you want something that feels like your dotfiles plus a thin layer on top, chezmoi can feel heavy.

yadm

Git, but smarter — with alternates, bootstrap hooks, and GPG encryption. Elegant if your workflow is already git-native. The tradeoff: encrypted files are all-or-nothing. You can't commit a .gitconfig and encrypt only the email field. And GPG key management adds its own maintenance burden.

GNU stow

Pure symlink management. Beautifully simple — a directory of packages, stow them, done. The tradeoff: zero secret handling. Your files go in git as-is. If that's fine for you, stow is probably the right answer. dotf is essentially stow with a template+secret layer bolted on and git sync built in.

Secret backends

Works with what you already use.

Reference secrets with a simple URI. dotf fetches them at sync time.

🔐
Proton Pass
Fetch from any Proton Pass vault via the CLI.
pass://vault/item/field
🔑
1Password
Pull secrets from 1Password using op CLI.
op://vault/item/field
🛡️
Bitwarden
Retrieve items from your Bitwarden vault.
bw://item-name/field
📦
Environment
Read directly from environment variables. Great for CI.
env://VAR_NAME
